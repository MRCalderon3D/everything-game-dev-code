# UI and UX

## Purpose
Define shared rules for clarity, usability, readability, interaction consistency, onboarding, and player feedback.

## Scope
Applies to HUD, menus, prompts, settings, inventories, shops, tutorials, overlays, meta layers, and in-world interaction feedback.

## UX Principles
- The interface should help players understand what they can do, what happened, and what matters next.
- Reduce friction on critical paths.
- Consistency beats novelty for recurring interactions.
- Feedback should be timely, legible, and proportional to player action.

## Information Architecture Rules
- Prioritize critical information by player context and urgency.
- Avoid overloading a single screen with mixed-priority information.
- Navigation structure should reflect player mental models, not internal team structure.
- Settings and accessibility options should be discoverable and categorized predictably.

## Interaction Rules
- Similar actions should behave similarly across the product.
- Interactive states must communicate focus, selection, success, failure, disabled state, and pending state.
- Confirmation prompts should be reserved for destructive or high-cost actions.
- Error messages must explain what happened and, when possible, what the player can do next.

## HUD and Feedback Rules
- HUD elements must justify their presence and update behavior.
- Avoid persistent noise that competes with critical moment-to-moment feedback.
- Combat, traversal, economy, and narrative systems should expose enough feedback for informed play without drowning the player.

## Onboarding Rules
- Tutorials should teach through action where possible.
- Do not front-load all information before it becomes relevant.
- Tutorial prompts and codex help should be revisitable when the game has complexity or long sessions between related mechanics.

## Accessibility and Input Rules
- UI must respect accessibility settings and input modality differences.
- Flows should be usable with all supported control schemes.
- Pointer, controller, keyboard, and touch interactions should not silently diverge in capability without deliberate design.

## Evaluation Rules
- UI and UX decisions should be validated through playtesting, heuristic review, or task-based observation.
- High-friction flows should be instrumented and reviewed.

## Deliverables
- UX goals.
- Information architecture and flow maps.
- Wireframes or UI specs.
- HUD inventory and behavior notes.
- Usability findings.

## Done Criteria
UI and UX work is ready when players can understand goals, navigate flows, and interpret feedback with minimal unnecessary friction.
